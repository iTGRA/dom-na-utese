Mobile-концепция
На mobile ≤ 767 px сайт превращается в вертикальный swipe-deck: каждый из 12 блоков — отдельный full-screen slide. Swipe вверх — следующий блок появляется с fade-in + slide-up. § 04 · Swipe
Mobile Swipe-архитектура
Техническая спецификация swipe-deck для mobile. Каждый блок — отдельный экран. Между экранами — fade-in + slide-up. Приложение-подобный опыт.
Концепция. На устройствах ≤ 767 px сайт работает не как привычный scroll, а как вертикальный карусель-deck. Пользователь свайпает вверх — следующий экран защёлкивается в viewport (scroll-snap mandatory). Каждый экран живёт сам по себе, с собственной типографикой, фоном и ритмом.
Это не авторский каприз. Для нашей аудитории (40–60 лет, частый просмотр с iPhone) это решает три задачи: 1) каждый блок получает 100% внимания без «проскроллил и не заметил»; 2) мы сохраняем редакционный ритм из desktop-версии в формате «страница за страницей»; 3) сайт ощущается как премиум-приложение, а не как лендинг.

Intersection Observer для появления

JS · СМЕНА ЭКРАНА 
// При каждом защёлкивании нового экрана
// запускаем staggered reveal типографики

const slides = document.querySelectorAll('.slide');

const io = new IntersectionObserver((entries) => {
  entries.forEach(entry => {
    if (entry.isIntersecting) {
      entry.target.classList.add('is-active');
      // обновляем progress-индикатор
      updateProgress(entry.target.dataset.index);
    } else {
      entry.target.classList.remove('is-active');
    }
  });
}, { threshold: 0.6 });

slides.forEach(s => io.observe(s));

Делать
  • Snap-точка всегда в начале экрана (scroll-snap-align: start)
  • Использовать 100svh / 100dvh, не 100vh — из-за Safari address bar
  • Отключать swipe-deck для tablet ≥ 768 — там обычный scroll
  • Обновлять URL (history.replaceState) с номером слайда для deep-linking
  • Поддерживать клавиатурную навигацию (Space / Arrow Down)
Не делать
  • Не блокировать нативный скролл полностью (Scroll-Jack)
  • Не превращать в horizontal swipe — только вертикальный
  • Не прятать индикаторы прогресса — они часть ориентации
  • Не ставить scroll-snap-stop: always без тестов на низких устройствах
  • Не забывать про prefers-reduced-motion (отключать snap!)

§ 06 · Motion
Моушн-система
Правила движения для категории. Всё должно быть медленно, плавно, «редакционно». Никакого bounce, никакого spring.
Философия. Сайт — это книга о доме. Анимация должна вести себя как бумага, как движение взгляда по странице: плавно, с паузами, без нервных колебаний. Ease-editorial — cubic-bezier(0.22, 1, 0.36, 1) — наш фирменный timing function. Он даёт тот самый «мягкий доезд» без ощущения механичности.

Правильно
  • Ease-editorial для всего крупного (блоки, модалки, открытие меню)
  • Ease-standard для мелкого (hover, button, input focus)
  • Prefers-reduced-motion: все transitions → 0ms, scroll-snap off
  • Параллакс только на desktop, и только для Hero и блока 11
  • Stagger для заголовков — 80–120ms между элементами, не больше
Неправильно
  • Spring / bounce / overshoot — категорически нет
  • Hover-эффекты на touch-устройствах (только active:)
  • Бесконечные loop-анимации фоном (только индикаторы hint)
  • Лоудеры-спиннеры вместо skeleton
  • Автовоспроизведение видео со звуком на Hero
  • Fade-out при закрытии модалки медленнее, чем open

§ 10 · Content
Тексты и микрокопия
Система текстовых элементов — от кнопок до ошибок форм. Всё пишется одним голосом.

Tone of voice

ГОЛОС БРЕНДА
Как пишем
  • Короткие предложения. Утвердительные.
  • Конкретные числа, даты, имена
  • Образные сравнения, вытекающие из архивов
  • Italic для ключевых слов в паре «факт + образ»
  • Тире как ритмический знак, не запятая
  • Прямая речь там, где уместна: «Приезжайте посмотреть»
Как не пишем
  • «Элитный», «премиум», «эксклюзивный» — слова-пустышки
  • Восклицательные знаки
  • «Мы рады предложить вам»
  • «Успейте», «не упустите», «только 3 лота»
  • Длинные составные предложения
  • Смайлы, эмодзи, ❗
